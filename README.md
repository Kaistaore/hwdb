1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД логистов?

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

## Задание 1. Выбор типов СУБД для предметных областей

### 1.1. Бюджетирование, фин. отчёты, прогнозирование рисков
**Тип СУБД:** Реляционная (SQL) – **PostgreSQL** или **Oracle**  
**Почему:**  
- Требуется целостность и чёткая структура (ACID, ссылочная целостность).  
- Сложные аналитические запросы (JOIN, оконные функции, агрегации).  
- Финансовые данные не терпят компромиссов по согласованности.

> **1.1.*** Хеширование долгое → использовать аппаратное ускорение: **Intel QAT**, **AES-NI**, **SHA extensions**, API `libcrypto`.

### 1.2. Лендинги (лиды) + CRM маркетологов и менеджеров
| Компонент | Рекомендуемая СУБД | Тип | Почему |
|-----------|-------------------|-----|--------|
| Лендинги | MongoDB | Документоориентированная | Гибкая схема, высокая скорость записи |
| CRM | MongoDB / PostgreSQL | Документная или реляционная | Быстрые запросы + гибкость статусов |

> **1.2.*** Можно закрыть одной СУБД: **MongoDB** с коллекциями `leads` и `customers`, используя Change Streams для синхронизации.

### 1.3. Корпоративная база знаний (нормы, правила, обучение)
**Тип СУБД:** Документоориентированная или иерархическая – **MongoDB** или **Elasticsearch**  
**Почему:** Простая вложенная структура (документы → разделы → компании). Легко моделировать дерево подразделений.

> **1.3.*** Да, можно использовать MongoDB из п.1.2: создать отдельную коллекцию `corporate_knowledge` с Materialized Path для иерархии.

### 1.4. Логистика (маршруты доставки, курьеры)
**Тип СУБД:** Графовая – **Neo4j** или **ArangoDB**  
**Почему:** Быстрая работа со связями – кратчайший путь, ближайший курьер, оптимизация развозки. На графовых БД такие запросы на порядки быстрее.

> **1.4.*** Отдел закупок: либо мультимодельная СУБД (ArangoDB – общая графовая + документная), либо отдельная реляционная БД + синхронизация через очереди (Kafka).

### 1.5.* Одна СУБД для всех задач
**Да, возможна при выборе мультимодельной СУБД:**  
- **ArangoDB** – документы + графы + транзакции.  
- **PostgreSQL + расширения** (JSONB, pgRouting) – но графовая производительность ниже.

**Лучший реальный подход:** гибрид – реляционная для финансов, графовая для логистики, документная для CRM/лендингов.

## Задание 2. Транзакции

### 2.1. Пополнение баланса телефона (6 шагов)
1. `BEGIN` – начало транзакции.  
2. Проверка авторизации и источника платежа.  
3. Блокировка средств на источнике (резервирование).  
4. Списание средств с источника → зачисление на счёт платёжного шлюза.  
5. Отправка команды оператору на увеличение баланса.  
6. `COMMIT` (при успехе) или `ROLLBACK` (при ошибке на шаге 5).

### 2.1.* Автоплатёж
1. Триггер по порогу баланса запускает автоплатёж.  
2. Проверка привязанной карты/счёта и согласия.  
3. Блокировка суммы на источнике.  
4. Списание и перевод оператору.  
5. Зачисление на баланс телефона.  
6. Логирование и уведомление. При неудаче – повторы с задержкой (retry policy) + компенсация по тайм-ауту.

## Задание 3. SQL vs NoSQL

### 3.1. Пять преимуществ SQL перед NoSQL
| # | Преимущество | Пояснение |
|---|--------------|-----------|
| 1 | Стандартизация | Единый язык SQL, предсказуемая схема |
| 2 | Сложные запросы и JOIN | Аналитика, отчёты, связывание таблиц |
| 3 | ACID-транзакции | Гарантии в распределённом окружении (классические SQL) |
| 4 | Целостность данных | Внешние ключи, ограничения, триггеры |
| 5 | Зрелая экосистема | BI-инструменты, ORM, бэкапы, репликация |

### 3.1.* Преимущества NewSQL перед SQL и NoSQL
| Перед SQL | Перед NoSQL |
|------------|--------------|
| Горизонтальное масштабирование «из коробки» | Полная поддержка ACID и сложных транзакций |
| Выше производительность на OLTP-кластере | SQL-интерфейс (низкий порог входа) |
| Автоматическое шардирование | Согласованность + масштабирование без жертв |

**Примеры NewSQL:** CockroachDB, YugabyteDB, TiDB, Google Spanner.

## Задание 4. Кластеры (1000 машин, огромные данные)

### Критерии выбора типа СУБД
- **Модель связей данных:** независимые vs сильно связанные.  
- **Согласованность:** строгая (финансы) vs приближённая (аналитика).  
- **Тип вычислений:** MapReduce / агрегации / графовые / потоковые.

### Лучшая модель распределённых вычислений
**Массово-параллельная обработка (MPP) с архитектурой Shared Nothing.**

| Компонент | Решение | Почему |
|-----------|---------|--------|
| Вычислительный движок | **Apache Spark** | Отказоустойчивость (RDD, lineage), работа с памятью/диском |
| Хранение | **TiDB** / **ClickHouse** / **Greenplum** | NewSQL + аналитика, горизонтальное масштабирование |

**Итоговый критерий:** соотношение объёма перемещаемых данных к вычислительной сложности → выбираем MPP с локальностью данных.

## Выводы
- **Не всё можно (и нужно) решать одной СУБД** – гибридные архитектуры эффективнее.  
- **SQL + графовые + документные** покрывают разные требования: целостность, связи, гибкость.  
- **NewSQL** – компромисс для масштабируемых OLTP с транзакциями.  
- **MPP + Spark** – стандарт для 1000+ узлов при больших объёмах данных.
